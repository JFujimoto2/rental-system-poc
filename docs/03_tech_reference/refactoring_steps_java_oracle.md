
# Java / Oracle システム リファクタリング実行ステップまとめ

## 前提認識
- Java / Oracle は技術的に問題ない
- 問題は「ブラックボックス化」と「判断できない状態」
- 目的は **フルリプレースではなく、段階的改善と保守費削減**

---

## 全体ロードマップ概要

1. 現状可視化（分かる状態を作る）
2. リスク低減（壊れない状態を作る）
3. 小さな改善（触っても大丈夫を証明）
4. 構造改善（保守費削減）
5. 必要な部分だけモダン化・クラウド化

---

## Step 1：現状可視化（0〜2か月）
**目的：判断材料を揃える。まだ削減しない。**

### 実施内容
- システム構成把握
  - Java / Oracle バージョン
  - サーバ構成・環境差異
- 業務・機能整理
  - 画面数
  - バッチ処理一覧
- DB理解
  - テーブル関係
  - 主なSQL・PL/SQL
- 運用・保守内容の整理
  - 障害履歴（直近1〜2年）
  - 夜間・休日対応実態
- ライセンス確認
  - Oracleエディション
  - オプション利用有無

### 成果物
- 構成図（As-Is）
- 危険箇所・触らなくてよい箇所の整理
- 「分かる人が1人いる」状態

---

## Step 2：リスク低減（1〜3か月）
**目的：壊さずに触れる土台を作る。**

### 実施内容
- ビルド・デプロイ整理
- テスト導入
  - 重要バッチ・業務ロジック優先
- 環境差異の解消
  - 検証 ≒ 本番

### ポイント
- 機能追加はしない
- 安定性最優先

---

## Step 3：小さな改善（2〜6か月）
**目的：成果を出し、信頼を得る。**

### 実施内容
- SQLチューニング
- バッチ処理時間短縮
- 不要な監視・運用削減
- Javaコードの軽微なリファクタ

### 成果
- 性能改善
- 障害・問い合わせ減少
- 保守工数削減の根拠ができる

---

## Step 4：保守体制・コスト見直し（6か月〜）
**目的：数字としての成果を出す。**

### 実施内容
- 保守契約見直し
  - 人月削減
  - 夜間待機縮小
- 内製／スポット対応検討
- ベンダー依存度低下

### 期待効果
- 月800万 → 500万以下を目指す

---

## Step 5：段階的モダン化・クラウド化（余力が出てから）
**目的：長期的な持続性確保。**

### 実施内容
- クラウド移行（Lift & Improve）
- 切り出しやすい機能のSaaS化
- Java / Oracle の段階的バージョンアップ

### 注意点
- 全面移行はしない
- コストとリスクを常に比較

---

## 全体を通した重要方針
- いきなり変えない
- 技術を否定しない
- 怖さではなく事実で判断
- 小さく始めて、確実に進める

---

## まとめ
このステップは
- 失敗しにくい
- 説明しやすい
- コスト削減に直結する

Java / Oracle 基幹を
**「触れない遺産」から「コントロールできる資産」へ変えるための実行計画。
